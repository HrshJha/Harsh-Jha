## 2. Content Principles

### Writing Style

- Minimal.
- Technical.
- Professional.
- Calm.
- Intentional.
- Evidence-first.
- Engineering-first.
- Concise enough for recruiters to scan.
- Specific enough for hiring managers and research engineers to evaluate.

### Voice

- Direct.
- Clear.
- Credible.
- Product-oriented.
- Research-aware.
- Not self-promotional.

### Tone

- Premium without sounding inflated.
- Technical without being dense.
- Confident without exaggeration.
- Focused on systems, architecture, research, and execution.

### Grammar Rules

- Use active voice where possible.
- Use complete sentences for paragraphs.
- Use fragments only for labels, tags, headings, and compact UI text.
- Do not use exclamation marks.
- Do not use inflated adjectives unless they are present in `FOUNDATION.md`.
- Do not imply completion for projects marked "In Progress".

### Formatting Rules

- Use sentence case for paragraphs.
- Use title case for primary navigation labels and major page headings.
- Use concise labels for buttons and links.
- Use bullets for scan-heavy technical content.
- Do not use skill percentages, ratings, progress bars, or proficiency meters.
- Do not use testimonials, fake metrics, visitor counters, blog labels, certifications sections, or GitHub contribution graph content.

### Terminology Rules

- Prefer "AI systems" over "AI apps" when describing the overall portfolio mission.
- Prefer "projects" over "work samples".
- Prefer "engineering decisions" over "design choices" when discussing technical proof.
- Prefer "in progress" only when tied to documented project status.
- Do not call project work "production-grade" unless referring to the mission statement: "Demonstrate the ability to engineer production-grade AI systems."
- Do not describe any project as launched, deployed, used by users, validated, or production-ready unless future source documents provide that evidence.

### Things Never To Say

The website must never use:

- Passionate
- Hardworking
- Quick learner
- Results-driven

The website must also avoid:

- Skill percentages.
- Visitor counters.
- Fake metrics.
- Fake testimonials.
- Blog.
- Certifications section.
- GitHub contribution graph.
- Unsupported achievements.
- Unsupported business claims.

### Things Always To Prefer

- Evidence over claims.
- Engineering over aesthetics.
- Projects over biography.
- Technical depth over generic description.
- Clear status labels over implied completion.
- Direct contact links over backend-dependent forms.
- Documented facts over inferred copy.
